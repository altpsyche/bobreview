# BobReview v1.0.7 Release Notes

**Release Date**: December 2024  
**Theme**: Plugin System

---

## Overview

BobReview v1.0.7 introduces a fully modular plugin system with focused architecture following SOLID and DRY principles. This release improves maintainability, testability, and predictability by eliminating backward compatibility layers and introducing focused interfaces.

---

## Key Features

### Focused Registries (Interface Segregation Principle)

PluginRegistry split into 11 focused registries, each handling a single type of component:

```python
# Old way (removed)
registry.register_theme(theme)
theme = registry.get_theme('dark')

# New way (focused)
registry.themes.register(theme)
theme = registry.themes.get('dark')
```

Available Registries:
- `registry.themes` - Theme registration
- `registry.llm_generators` - LLM generator registration
- `registry.data_parsers` - Data parser registration
- `registry.widgets` - Widget registration
- `registry.chart_types` - Chart type registration
- `registry.pages` - Page registration
- `registry.services` - Service registration
- `registry.report_systems` - Report system registration
- `registry.chart_generators` - Chart generator registration
- `registry.context_builders` - Context builder registration
- `registry.template_paths` - Template path registration

### Focused Config Classes (Interface Segregation Principle)

ReportConfig split into 5 focused config classes, each handling a single concern:

```python
# Old way (removed)
config.draw_soft_cap = 600
config.llm_provider = 'openai'
config.dry_run = True

# New way (focused)
config.thresholds.draw_soft_cap = 600
config.llm.provider = 'openai'
config.execution.dry_run = True
```

Available Config Classes:
- `config.thresholds` - Performance thresholds (draw_soft_cap, tri_hard_cap, etc.)
- `config.llm` - LLM settings (provider, model, temperature, etc.)
- `config.execution` - Execution behavior (dry_run, verbose, sample_size, etc.)
- `config.output` - Output settings (embed_images, theme_id, disabled_pages, etc.)
- `config.cache` - Cache settings (cache_dir, use_cache, clear_cache)

### Dependency Injection (Dependency Inversion Principle)

ReportSystemExecutor now accepts dependencies via constructor:

```python
# Old way (global singletons)
executor = ReportSystemExecutor(system_def, config)
# Internally uses: get_container(), get_registry(), get_template_engine()

# New way (dependency injection)
executor = ReportSystemExecutor(
    system_def, config,
    container=container,
    registry=registry,
    template_engine=template_engine
)
```

Benefits:
- Easier to test (can inject mocks)
- Better parallelization (no shared global state)
- Follows Dependency Inversion Principle

### Single Responsibility Classes

ReportSystemExecutor split into focused responsibility classes:

- ConfigMerger - Handles configuration merging only
- ServiceValidator - Validates required services only
- PluginLifecycleManager - Manages plugin lifecycle hooks only
- ReportSystemExecutor - Orchestrates report generation only

### DRY Utilities

Common patterns extracted to reusable utilities:

- safe_plugin_call() - Consistent error handling for plugin methods
- merge_config() - Reusable configuration merging
- call_plugin_lifecycle_hooks() - Plugin lifecycle management

### For Developers

- Clearer intent: Focused interfaces make code purpose obvious
